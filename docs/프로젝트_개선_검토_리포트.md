각 항목에는 **중요도**(🔴 높음 / 🟡 중간 / 🟢 낮음)와 **구현 난이도**(⭐~⭐⭐⭐)를 표기했습니다.

## 1. 모델 아키텍처 개선

### 🔴 1-1. Attention에 Dropout이 없음 (중요도: 높음 / 난이도: ⭐)

**현재 상태**: `model/attention.py`에 어텐션 확률 분포(probs)와 출력 투사(wo) 어디에도 Dropout이 없습니다.

**문제점**: FineWeb-edu 100K 문서 수준의 데이터로 162M 파라미터를 학습하면, 과적합(Overfitting)이 거의 확실하게 발생합니다. Dropout은 과적합 방지를 위한 가장 기본적인 정규화 기법입니다.

**개선 방향**:
```python
# attention.py forward() 내부
self.attn_dropout = nn.Dropout(p=0.1)
self.resid_dropout = nn.Dropout(p=0.1)

probs = self.attn_dropout(torch.softmax(scores, dim=-1))  # 어텐션 확률에 적용
return self.resid_dropout(self.wo(output))                  # 잔차 연결 전 적용
```

> ⚠️ FFN 출력에도 동일하게 Dropout을 추가해야 합니다. `model/ffn.py`의 SwiGLU 출력, `model/moe_ffn.py`의 최종 출력 모두 해당됩니다.

---

### 🟡 1-2. Embedding 가중치 공유(Weight Tying) 옵션 추가 (중요도: 중간 / 난이도: ⭐)

**현재 상태**: `model/moe_transformer.py` L43~L66에서 `token_embeddings`와 `lm_head`가 별도 가중치를 사용합니다(untied).

**문제점**: Untied 구조는 파라미터가 약 24.6M 추가되어 162M이 되지만, 데이터 규모가 작을 때는 이 추가 파라미터가 오히려 과적합을 악화시킵니다. GPT-2, LLaMA 초기 모델들은 데이터가 충분해질 때까지 Weight Tying을 사용했습니다.

**개선 방향**:
```python
# 생성자에 tie_embeddings 옵션 추가
if tie_embeddings:
    self.lm_head.weight = self.token_embeddings.weight  # 가중치 공유
```

---

### 🟡 1-3. Shared Expert 도입 (DeepSeek-MoE 스타일) (중요도: 중간 / 난이도: ⭐⭐)

**현재 상태**: 청사진에도 "Shared Expert: 없음 (추후 확장 고려)" 로 명시되어 있습니다.

**문제점**: 현재 구조에서는 모든 전문가가 라우팅에 의존하므로, 범용적인 언어 지식(문법, 접속사, 관사 등)도 특정 전문가에 편중될 수 있습니다.

**개선 방향**: MoE 블록마다 **1개의 Shared Expert**를 추가하여, 라우터를 거치지 않고 모든 토큰이 반드시 통과하는 공용 FFN을 배치합니다.

```python
# moe_ffn.py 에 shared expert 추가
self.shared_expert = ExpertFFN(d_model, d_ff)

# forward() 에서
shared_out = self.shared_expert(flat_x)
out = shared_out + routed_out  # 공용 + 라우팅 결과 합산
```

---

### 🟢 1-4. GQA (Grouped Query Attention) 적용 (중요도: 낮음 / 난이도: ⭐⭐)

**현재 상태**: `model/attention.py`에서 Q, K, V 모두 동일한 개수(8개)의 헤드를 사용하는 MHA(Multi-Head Attention) 구조입니다.

**문제점**: 추론 시 KV Cache 메모리가 헤드 수에 비례하여 커집니다. 162M 규모에서는 큰 이슈가 아니지만, 향후 규모를 키울 때 병목이 됩니다.

**개선 방향**: K, V의 헤드 수를 Q 대비 줄여 (예: K, V = 2헤드, Q = 8헤드) 메모리를 절약하면서 성능 저하를 최소화할 수 있습니다. LLaMA 2 이후 모든 주요 모델들이 채택한 기법입니다.

## 2. 학습 파이프라인 개선

### 🔴 2-1. Gradient Accumulation 미지원 (중요도: 높음 / 난이도: ⭐)

**현재 상태**: `train/train.py` L199~L222에서 매 배치마다 즉시 `optimizer.step()`을 호출합니다.

**문제점**: Colab A100에서 batch_size=32, seq_len=1024로 학습하면 GPU 메모리(40GB)가 빠듯합니다. Gradient Accumulation 없이는 유효 배치 크기를 키울 수 없어, 학습 안정성과 수렴 품질이 제한됩니다.

**개선 방향**:
```python
# 예: 4회 누적 -> 유효 배치 128
gradient_accumulation_steps = 4

for step, batch in enumerate(dataloader):
    with accelerator.accumulate(model):
        logits, loss, ... = model(input_ids, labels)
        accelerator.backward(loss / gradient_accumulation_steps)
        
        if (step + 1) % gradient_accumulation_steps == 0:
            accelerator.clip_grad_norm_(model.parameters(), max_norm=1.0)
            optimizer.step()
            scheduler.step()
            optimizer.zero_grad()
```

---

### 🔴 2-2. Validation Loss 주기적 측정 누락 (중요도: 높음 / 난이도: ⭐)

**현재 상태**: `train/train.py`에서 학습 루프 중간에 Validation Loss를 측정하는 코드가 전혀 없습니다. Validation은 학습 완료 후 `train/evaluate.py`를 별도로 실행해야만 확인 가능합니다.

**문제점**: 과적합이 시작되는 시점을 실시간으로 감지할 수 없습니다. 5000 스텝 학습을 다 끝낸 후에야 과적합을 발견하면 시간과 GPU 비용이 낭비됩니다.

**개선 방향**: 500 스텝마다 `model.eval()` -> `torch.no_grad()` 상태에서 val 데이터 일부를 돌려 Validation PPL을 로그에 함께 기록합니다. Train Loss와 Val Loss의 격차가 벌어지기 시작하면 Early Stopping을 고려할 수 있습니다.

---

### 🟡 2-3. 전문가 카운팅 루프 비효율 (중요도: 중간 / 난이도: ⭐)

**현재 상태**: `train/train.py` L66~L69와 `train/evaluate.py` L154~L156에서 전문가 카운팅을 Python for 루프로 수행합니다.

```python
# 현재: 느린 Python 루프
for idx in all_indices.view(-1):
    if idx.item() < 4:
        counts[idx.item()] += 1
```

**문제점**: 토큰 수가 수십만 개가 되면 이 루프가 학습 속도를 유의미하게 저하시킵니다.

**개선 방향**:
```python
# 개선: PyTorch 벡터 연산으로 즉시 카운팅 (수백 배 빠름)
counts = torch.bincount(all_indices.view(-1), minlength=4).float()
```

---

### 🟡 2-4. Learning Rate 스케줄러에 min_lr 미설정 (중요도: 중간 / 난이도: ⭐)

**현재 상태**: `train/train.py` L153~L159의 코사인 감쇄 스케줄러가 학습 후반에 lr -> 0에 수렴합니다.

**문제점**: 학습률이 완전히 0에 도달하면 학습 후반부에서 가중치 업데이트가 사실상 정지됩니다. 이는 최종 수렴 품질을 떨어뜨릴 수 있습니다.

**개선 방향**: 최소 학습률(min_lr)을 피크 lr의 10% 수준으로 설정하여 바닥을 깔아줍니다.
```python
min_lr = args.lr * 0.1
return max(min_lr / args.lr, 0.5 * (1.0 + np.cos(np.pi * progress)))
```

## 3. 데이터 처리 개선

### 🟡 3-1. 데이터 셔플링 품질 (중요도: 중간 / 난이도: ⭐)

**현재 상태**: `train/prepare_data.py`에서 전체 토큰 스트림을 순서대로 block_size 단위로 자른 후 train/val로 분할합니다.

**문제점**: FineWeb-edu에서 문서가 특정 주제 순서로 정렬되어 있다면, Train의 마지막 10%(Val 직전 구간)과 Val이 비슷한 주제의 문서로 편중될 수 있습니다. 또한 블록 단위 셔플이 없으므로 같은 문서 내 인접 블록이 연속으로 학습에 투입됩니다.

**개선 방향**:
```python
# 블록 생성 후 분할 전에 전체 셔플
np.random.shuffle(blocks)
train_blocks = blocks[:split_idx]
val_blocks = blocks[split_idx:]
```

---

### 🟢 3-2. 토큰 수 기반 동적 배치 사이즈 (중요도: 낮음 / 난이도: ⭐⭐)

**현재 상태**: 고정 block_size(1024) x 고정 batch_size(32)로 학습합니다.

**문제점**: 모든 문서를 1024로 잘라붙이므로 문서 경계에서 맥락이 끊기는 정보 손실이 발생합니다.

**개선 방향**: 문서 단위로 자연스럽게 패킹(Document Packing)하여 가변 길이를 유지하되, 배치 내 총 토큰 수를 일정하게 맞추는 Dynamic Batching을 적용할 수 있습니다.

## 4. 추론 및 서빙 개선

### 🔴 4-1. 텍스트 생성(Inference) 스크립트 부재 (중요도: 높음 / 난이도: ⭐⭐)

**현재 상태**: 학습된 체크포인트로 실제 텍스트를 생성(Autoregressive Generation)하는 코드가 없습니다.

**문제점**: 학습이 끝나도 모델이 실제로 어떤 문장을 생성하는지 확인할 방법이 없습니다. PPL 수치만으로는 모델 품질을 직관적으로 체감하기 어렵습니다.

### 🟡 4-2. KV Cache 미구현 (중요도: 중간 / 난이도: ⭐⭐)

**현재 상태**: `model/attention.py`에서 매번 전체 시퀀스에 대해 Q, K, V를 처음부터 다시 계산합니다.

**문제점**: Autoregressive 생성 시 이전 시점의 K, V를 캐싱하지 않으면 토큰을 하나 생성할 때마다 전체 시퀀스를 재연산해야 하므로, 생성 속도가 O(T^2)이 됩니다.

**개선 방향**: Attention 모듈에 `past_key_values` 매개변수를 추가하여, 이전 시점의 K, V 텐서를 캐싱하고 새 토큰의 K, V만 추가로 계산합니다.

## 6. 운영 안정성 개선

### 🔴 6-1. 체크포인트 무한 누적 (중요도: 높음 / 난이도: ⭐)

**현재 상태**: `train/utils.py` save_checkpoint() 함수에서 1000 스텝마다 체크포인트를 저장하지만, 이전 체크포인트를 삭제하지 않습니다.

**문제점**: 162M 모델의 체크포인트 1개 약 650MB~1.2GB입니다. 5000 스텝 학습 시 최소 5개 = ~5GB 이상의 체크포인트가 Google Drive에 누적됩니다. 여러 실험을 반복하면 Drive 15GB 무료 한도를 금방 초과합니다.

**개선 방향**: 최근 N개(예: 3개)만 유지하고 오래된 체크포인트를 자동 삭제하는 로테이션 로직을 추가합니다.
```python
def cleanup_old_checkpoints(ckpt_dir, pattern, keep_last=3):
    ckpts = sorted(glob.glob(os.path.join(ckpt_dir, f"{pattern}*.pt")), key=os.path.getmtime)
    for old_ckpt in ckpts[:-keep_last]:
        os.remove(old_ckpt)
```
