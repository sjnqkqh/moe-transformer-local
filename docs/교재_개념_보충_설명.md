## 1. KV Cache

### 역할

모델이 문장을 한 단어씩 생성할 때, 이전에 계산한 Key·Value 행렬을 **버리지 않고 저장(캐시)해두는** 기법입니다.

Attention 계산에서 현재 단어의 Query는 지금 막 생성된 것이지만, Key와 Value는 이전에 이미 계산한 것과 동일합니다. 이것을 매번 다시 계산하는 것은 낭비입니다.

**예시:**  
"오늘 날씨가" 다음에 "좋다"를 생성할 때를 생각해보겠습니다.

- KV Cache 없음: "오늘", "날씨가", "좋다" 세 단어 모두 K·V 새로 계산
- KV Cache 있음: "오늘", "날씨가"의 K·V는 이미 저장되어 있으므로, "좋다"의 K·V만 새로 계산

생성할 단어 수가 늘어날수록 절감 효과가 커집니다. 1000 토큰을 생성한다면 이론적으로 연산량이 거의 절반으로 줄어듭니다.

### 사용 단계

```
학습(Training)   ← KV Cache 없음 (배치 전체를 한 번에 처리하므로 불필요)
추론(Inference)  ← KV Cache 있음 (한 단어씩 순차 생성)
```

### 이 프로젝트에 없는 이유

이 프로젝트는 **학습 전용**입니다. 학습할 때는 정답 문장 전체를 한꺼번에 모델에 넣어 병렬 처리하므로(Teacher Forcing 방식), 한 단어씩 순차 생성하는 상황 자체가 발생하지 않습니다. KV Cache는 실제 서비스 추론 단계에서 붙이는 기능입니다.

## 2. 디코딩 전략 (Greedy / Beam Search / Top-p Sampling)

### 역할

모델이 다음 단어의 확률 분포를 계산한 뒤, **그 분포에서 어떤 단어를 실제로 선택할지** 결정하는 규칙입니다.  
모델 자체는 "각 단어가 나올 확률"만 계산할 뿐, 어떤 전략으로 선택하느냐는 별개의 문제입니다.

**주요 전략:**

- **Greedy Decoding**: 매 단계에서 확률이 가장 높은 단어 하나를 선택합니다.  
  예: `[사과: 60%, 바나나: 30%, 포도: 10%]` → 항상 "사과" 선택  
  단점: 지금 가장 높은 단어를 고르면 전체 문장 품질이 낮아질 수 있습니다.

- **Beam Search**: 상위 k개 후보를 동시에 유지하며 탐색합니다.  
  예: k=3이면 "사과", "바나나", "포도" 세 경우를 모두 이어 생성해보고, 문장 전체 확률이 높은 것을 최종 선택합니다.  
  기계 번역처럼 정확한 출력이 필요한 태스크에 적합합니다.

- **Top-p Sampling (Nucleus Sampling)**: 누적 확률 합이 p(예: 0.9)를 넘는 후보군 안에서 무작위 선택합니다.  
  ChatGPT처럼 다양하고 자연스러운 문장을 생성할 때 사용합니다.

### 사용 단계

```
학습(Training)   ← 해당 없음
추론(Inference)  ← 모델이 단어를 생성할 때마다 적용
```

### 이 프로젝트에 없는 이유

학습할 때는 다음 단어를 "선택"하지 않습니다. 정답 문장을 통째로 넣고 손실(Loss)을 계산하기만 합니다. 디코딩 전략은 학습이 끝난 모델을 실제로 사용할 때 필요한 기능입니다.

## 3. 긴 문맥 처리

### 역할

이 프로젝트의 모델은 한 번에 최대 1024 토큰(단어)까지만 처리할 수 있습니다(`max_seq_len=1024`). 이보다 긴 문서나 대화를 처리하려면 별도의 기법이 필요합니다.

**주요 기법:**

- **Sliding Window**: 1024 토큰씩 잘라 겹치게 처리합니다.  
  예: 3000 토큰 문서를 [0~1024], [512~1536], [1024~2048] 식으로 분할.

- **RoPE 외삽(Extrapolation)**: 이 프로젝트도 RoPE를 쓰지만, 1024 이상의 위치에서는 학습한 적 없는 회전 각도가 나타나 성능이 저하됩니다. 이를 해결하는 변형 기법들(YaRN, LongRoPE 등)이 있습니다.

- **Flash Attention**: 긴 시퀀스에서 Attention 계산을 메모리 효율적으로 수행하는 알고리즘입니다. 길이가 길어질수록 Attention의 메모리 사용량이 제곱으로 늘어나는 문제를 해결합니다.

### 이 프로젝트에 없는 이유

162M 파라미터 규모의 학습 실험 목적 프로젝트에서 1024 토큰은 충분한 컨텍스트입니다. 긴 문맥 처리는 대형 모델을 실서비스에 올릴 때 필요한 엔지니어링 과제입니다.

## 4. 추론 최적화 / 모델 서빙

### 역할

학습된 모델을 실제 사용자 요청에 빠르게 응답하도록 최적화하는 기술들입니다.

**주요 기법:**

- **양자화(Quantization)**: 모델 가중치를 32비트 소수 대신 8비트나 4비트 정수로 저장합니다.  
  예: 162M 모델은 FP32 기준 약 650MB인데, INT8 양자화 시 약 160MB로 줄어듭니다.

- **모델 컴파일**: PyTorch의 `torch.compile()`을 이용해 연산 그래프를 최적화합니다.

- **배치 처리(Batching)**: 여러 사용자 요청을 묶어 GPU에서 한 번에 처리합니다.

- **서빙 프레임워크**: vLLM, TGI(Text Generation Inference) 등 LLM 전용 서버를 사용합니다.

### 이 프로젝트에 없는 이유

이 프로젝트의 목적은 **모델 구조를 이해하고 학습시키는 것**입니다. 서빙 최적화는 학습이 완료된 이후, 실제 서비스 배포 단계에서 별도로 다루는 영역입니다.

## 5. Fine-tuning (태스크별 적응 학습)

### 역할

대규모 텍스트로 사전학습(Pre-training)된 모델을 특정 목적에 맞게 **추가 학습**하는 과정입니다.

사전학습: 인터넷의 방대한 텍스트로 언어 자체를 익힘 → 다음 단어 예측  
Fine-tuning: 그 위에 특정 데이터로 짧게 추가 학습 → 특정 태스크 수행

**예시:**

```
사전학습 모델 (일반 언어 이해)
    ↓ Fine-tuning
    ├── 감성 분석 모델 (영화 리뷰 데이터로 학습)
    ├── 번역 모델 (영한 번역 데이터로 학습)
    └── 의료 상담 모델 (의료 Q&A 데이터로 학습)
```

Fine-tuning은 처음부터 학습하는 것보다 훨씬 적은 데이터와 시간으로 원하는 성능을 얻을 수 있습니다.

### 이 프로젝트에 없는 이유

이 프로젝트는 아직 사전학습 단계입니다. Fine-tuning은 사전학습이 완료된 모델이 있어야 가능한 다음 단계입니다. FineWeb-edu 데이터로 사전학습이 끝나면, 그 체크포인트를 이용해 Fine-tuning을 진행하는 코드를 추가할 수 있습니다.

## 6. Prompting / Instruction Tuning

### 역할

**Prompting**은 학습된 모델에게 지시문(프롬프트)을 입력해 원하는 출력을 이끌어내는 기술입니다.  
모델 가중치를 바꾸지 않고, **입력 텍스트만 조작**합니다.

```
나쁜 프롬프트: "번역해"
좋은 프롬프트: "다음 한국어 문장을 자연스러운 영어로 번역하세요. 한국어: 오늘 날씨가 좋습니다. 영어:"
```

**Instruction Tuning**은 "지시를 따르는 능력" 자체를 모델에 학습시키는 Fine-tuning의 한 형태입니다.

```
학습 데이터 형식 예시:
  지시: "다음 문장을 요약하세요."
  입력: "오늘 서울에서는 ... (긴 기사)"
  출력: "서울에서 ... 발생했다."
```

이렇게 지시-입력-출력 쌍을 수만 개 학습시키면, 새로운 지시도 따를 수 있는 모델이 됩니다.

### 이 프로젝트에 없는 이유

Instruction Tuning은 Fine-tuning의 일종이므로 사전학습 이후 단계입니다. 또한 이 모델은 현재 162M 파라미터 수준이라, 지시를 안정적으로 따르려면 더 큰 규모와 전용 데이터가 필요합니다.

## 7. RLHF (Reinforcement Learning from Human Feedback)

### 역할

모델이 "사람이 좋아하는 방향"으로 출력을 생성하도록 **강화학습으로 조정**하는 기법입니다.  
ChatGPT를 단순한 텍스트 예측 모델에서 대화에 유용한 어시스턴트로 만드는 핵심 과정입니다.

**과정:**

```
1. 모델이 같은 질문에 여러 답변 생성
       예: "파이썬이란 무엇인가?"
       답변A: "파이썬은 프로그래밍 언어입니다. 읽기 쉽고..."
       답변B: "파이썬은 뱀의 종류입니다." (엉뚱한 답변)

2. 사람이 A > B 순위를 매김

3. 순위 데이터로 "보상 모델(Reward Model)" 학습
       보상 모델: 어떤 답변이 더 좋은지 점수 매기는 별도 모델

4. 보상 모델의 점수를 높이는 방향으로 원래 모델을 강화학습으로 조정
```

### 이 프로젝트에 없는 이유

RLHF는 사전학습 → Instruction Tuning이 완료된 모델에 적용하는 마지막 단계입니다. 또한 사람 레이블러, 보상 모델 학습, 강화학습 루프 구축 등 상당한 인프라가 필요해 개인 프로젝트 범위를 벗어납니다.

## 8. 인코더 전용 모델 (BERT / KoBERT / ALBERT)

### 역할

BERT는 이 프로젝트와 반대 방향의 구조입니다. **텍스트를 생성**하는 게 아니라, **텍스트를 이해(분류)**하는 데 특화된 모델입니다.

| | BERT 계열 (인코더) | 이 프로젝트 (디코더) |
|---|---|---|
| 방향 | 양방향 — 앞뒤 단어 모두 참고 | 단방향 — 이전 단어만 참고 |
| Attention Masking | 없음 (전체 참조 가능) | Causal Mask (미래 차단) |
| 주 사용처 | 분류, 감성 분석, NER | 텍스트 생성, 대화 |

**예시:**

```
BERT: "이 영화는 정말 [MASK] 영화다" → [MASK] = "재미있는" (빈칸 채우기 학습)
GPT:  "오늘 날씨가" → "좋다" (다음 단어 예측 학습)
```

**KoBERT**: 한국어 텍스트로 학습한 BERT입니다.  
**ALBERT**: BERT의 파라미터를 공유 방식으로 줄여 경량화한 버전입니다.

### 이 프로젝트에 없는 이유

이 프로젝트는 Decoder-only 구조입니다. BERT를 구현하려면 Causal Mask를 제거하고, 학습 목적(다음 단어 예측 대신 빈칸 채우기)을 바꿔야 합니다. 목적 자체가 다른 별도의 모델 패밀리입니다.

## 9. 인코더-디코더 모델 (T5)

### 역할

인코더와 디코더를 **모두 가진** 구조입니다. 입력 텍스트를 인코더로 이해하고, 그 정보를 바탕으로 디코더가 출력을 생성합니다.

```
입력: "영어로 번역: 오늘 날씨가 맑습니다"
인코더 처리 → 문장 의미 벡터 추출
디코더 생성 → "The weather is clear today"
```

T5는 모든 NLP 태스크를 "텍스트 입력 → 텍스트 출력" 형식으로 통일해서 학습합니다.

```
번역: "translate Korean to English: ..."
요약: "summarize: ..."
분류: "sentiment: ..."
```

### 이 프로젝트에 없는 이유

T5 구조를 구현하려면 인코더 스택과 인코더-디코더 크로스 어텐션이 추가로 필요합니다. 이 프로젝트는 GPT 계열(Decoder-only)을 목표로 설계되었으므로 해당 구조가 없습니다.

## 10. 인코더-디코더 어텐션

### 역할

T5, 원본 Transformer(번역 모델)처럼 인코더와 디코더가 함께 있는 구조에서만 나타나는 어텐션입니다.

디코더 블록 내부에 Self-Attention 외에 **하나가 더** 존재합니다:

```
디코더 블록 구성 (T5/원본 Transformer):
  1. Self-Attention: 지금까지 생성한 단어들끼리 참조
  2. 인코더-디코더 Attention: 인코더가 만든 벡터를 참조  ← 이것
  3. FFN
```

2번 어텐션에서 Query는 디코더에서 오고, Key·Value는 인코더 출력에서 옵니다.  
"지금 내가 생성하는 단어(Query)와 입력 문장(Key·Value) 중 어느 부분이 관련 있는가"를 계산합니다.

### 이 프로젝트에 없는 이유

인코더가 없으므로 이 어텐션도 필요 없습니다. Decoder-only 모델은 인코더 출력을 참조할 필요가 없기 때문입니다.

## 11. LLM / 스케일링 법칙

### 역할

**스케일링 법칙(Scaling Laws)**은 모델 크기, 데이터 양, 연산량이 증가할수록 성능이 어떻게 향상되는지를 수식으로 나타낸 경험적 법칙입니다.

2020년 OpenAI가 발표한 연구에 따르면, 세 가지 요소를 동시에 늘릴 때 성능이 예측 가능하게 향상됩니다:

```
파라미터 수 (N)
학습 데이터 양 (D)
연산량 (C = 6ND 로 근사)
```

**Chinchilla 법칙(2022)**: 최적 학습을 위해 파라미터 1개당 학습 토큰 20개가 필요합니다.  
- 162M 파라미터 → 약 32억 토큰 필요

이 법칙 덕분에 GPT-4 같은 대형 모델을 만들기 전에 소형 모델 실험으로 성능을 예측할 수 있습니다.

### 이 프로젝트에 없는 이유

스케일링 법칙은 실험 설계와 분석의 개념적 틀이지, 코드로 구현하는 것이 아닙니다. 이 프로젝트는 스케일링 법칙을 **적용받는** 대상(162M 규모 모델)입니다.

## 12. Vision Transformer / 멀티모달 LLM

### 역할

**Vision Transformer (ViT)**는 이미지를 처리하는 데 Transformer 구조를 적용한 모델입니다.

이미지를 16×16 픽셀 패치로 잘라, 각 패치를 하나의 "단어"처럼 취급해 임베딩합니다.

```
이미지 224×224 픽셀
  → 14×14 = 196개 패치로 분할
  → 각 패치를 벡터로 임베딩
  → Transformer로 처리
  → 이미지 분류 결과
```

**멀티모달 LLM**은 텍스트와 이미지를 함께 처리할 수 있는 모델입니다.

```
입력: [이미지: 고양이 사진] + "이 사진에서 무엇이 보이나요?"
출력: "주황색 고양이가 소파 위에 앉아 있습니다."
```

GPT-4V, Gemini, Claude가 이 방식을 사용합니다.

### 이 프로젝트에 없는 이유

이 프로젝트는 텍스트 전용 언어 모델입니다. 이미지 처리를 추가하려면 이미지 인코더(ViT 등)와 텍스트 모달리티를 연결하는 별도 설계가 필요합니다.

---

## 13. 모델 평가 / 벤치마크

### 역할

학습된 모델의 성능을 객관적으로 측정하는 방법들입니다.

이 프로젝트에는 Perplexity(당혹도)만 구현되어 있습니다([`train/evaluate.py`](../train/evaluate.py)).

**Perplexity**: 모델이 다음 단어를 얼마나 확신하는지 나타내는 지표입니다.  
- 낮을수록 좋음. 완벽한 모델은 1에 가까움  
- 예: PPL=50이면 평균적으로 50개 후보 중 하나를 고르는 수준의 불확실성

**교재에 등장하는 주요 벤치마크:**

| 벤치마크 | 측정 내용 | 예시 태스크 |
|----------|----------|-------------|
| MMLU | 전문 지식 이해 | 의학, 법학, 수학 객관식 |
| HellaSwag | 상식 추론 | 문장 이어쓰기 선택 |
| HumanEval | 코드 생성 | 함수 구현 |
| KoLU | 한국어 이해 | 한국어 버전 MMLU |

### 이 프로젝트에 부분적으로만 있는 이유

벤치마크 평가는 모델이 어느 정도 학습된 이후에 의미가 있습니다. 162M 파라미터에 제한된 학습 데이터로는 MMLU 같은 벤치마크에서 의미 있는 점수를 얻기 어렵습니다. Perplexity는 학습 진행 상황을 모니터링하기에 가장 간단하고 직접적인 지표이므로 현 단계에서는 충분합니다.
